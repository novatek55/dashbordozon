# План: ежедневный сбор истории остатков

## Зачем это нужно

Сейчас график остатков в `Диагностика SKU` считается обратным методом:
- берем текущий `FBO`
- берем текущий `FBS`
- идем назад по дням
- прибавляем дневные продажи по схемам `FBO` и `FBS/RFBS`

Это хороший временный вариант, но он не является фактической историей склада.
Чтобы график остатков стал точным, нужно каждый день сохранять снапшот остатков с датой и временем.

## Что уже есть в проекте

Текущие остатки уже доступны из двух источников:

### FBO

Источник: `analytics_stocks`

Используемые поля:
- `sku`
- `offer_id`
- `warehouse_id`
- `warehouse_name`
- `cluster_id`
- `cluster_name`
- `available_stock_count`
- `requested_stock_count`
- `transit_stock_count`
- `waiting_docs_stock_count`
- `last_synced_at`

Логика:
- фактический срез `FBO` = сумма по SKU
- можно хранить как общий `fbo_total`
- можно хранить и детализацию:
  - `fbo_available`
  - `fbo_supply`
  - `fbo_transit`
  - `fbo_acceptance`

### FBS

Источник: `fbs_warehouse_stocks`

Используемые поля:
- `sku`
- `offer_id`
- `warehouse_id`
- `warehouse_name`
- `present`
- `reserved`
- `last_synced_at`

Логика:
- фактический срез `FBS` = сумма `present` по SKU

## Что нужно сделать

Нужен отдельный скрипт ежедневного накопления снапшотов остатков.

### 1. Создать таблицу истории

Предлагаемая таблица: `stock_daily_snapshots`

Минимальные поля:
- `id`
- `snapshot_date`
- `snapshot_at`
- `sku`
- `offer_id`
- `warehouse_id`
- `warehouse_name`
- `stock_type`
- `cluster_id`
- `cluster_name`
- `stock_total`
- `stock_available`
- `stock_supply`
- `stock_transit`
- `stock_acceptance`
- `stock_reserved`
- `source_table`
- `created_at`

Рекомендации:
- `stock_type`: `FBO` или `FBS`
- `snapshot_date` хранить отдельно от `snapshot_at`
- сделать уникальность на уровне:
  - `snapshot_date`
  - `stock_type`
  - `sku`
  - `warehouse_id`
  - `cluster_id`

Это позволит безопасно перезапускать сбор в один и тот же день через `UPSERT`.

### 2. Реализовать скрипт сбора

Предлагаемое имя:
- `src/capture_stock_snapshot.py`

Что должен делать скрипт:
- подключаться к PostgreSQL
- читать актуальные данные из `analytics_stocks`
- читать актуальные данные из `fbs_warehouse_stocks`
- приводить строки к единому формату
- писать записи в `stock_daily_snapshots`
- работать идемпотентно для текущей даты

Логика записи:
- один запуск в день сохраняет один дневной срез
- если запускаем повторно в этот же день:
  - либо обновляем строки этого дня
  - либо не вставляем дубликаты

### 3. Подключить запуск по расписанию

Варианты запуска:

#### Вариант A. Windows Task Scheduler

Самый простой и надежный для текущего окружения Windows.

Сценарий:
- каждый день в заданное время
- запуск команды:

```powershell
E:\скрипты OZ\ozonapi\.venv\Scripts\python.exe -m src.capture_stock_snapshot
```

Рекомендуемое время:
- после основного утреннего обновления
- например `08:10` или `09:10` по Москве

#### Вариант B. Встроить в существующий sync-процесс

Можно встроить в кнопку `Обновить данные...`, но это хуже для накопления истории:
- если в какой-то день кнопку не нажмут, дневной срез пропадет
- если нажмут много раз, нужна дополнительная защита от дублей

Поэтому лучше:
- ежедневный `scheduler`
- кнопка обновления отдельно

### 4. Перевести график остатков на фактическую историю

После накопления истории график 5 должен брать данные уже из `stock_daily_snapshots`.

Новая логика графика:
- за 30 дней выбираем дневные срезы по SKU
- строим отдельно:
  - `FBO`
  - `FBS`
- при желании добавляем:
  - stacked bars
  - line chart
  - детализацию `available / transit / supply`

## Какой результат хотим получить

После внедрения:
- график остатков станет фактическим, а не расчетным
- можно будет показывать отдельно `FBO` и `FBS`
- можно будет анализировать:
  - реальные провалы по остаткам
  - зависание в транзите
  - рост supply/acceptance

## Минимальный MVP

Если делать быстро, достаточно первого этапа:

1. создать `stock_daily_snapshots`
2. сохранять один снимок в день
3. хранить только:
   - `snapshot_date`
   - `sku`
   - `offer_id`
   - `stock_type`
   - `stock_total`
4. перевести график 5 на эту таблицу

Этого уже хватит, чтобы график стал реальным.

## Следующий шаг

Следующим этапом можно реализовать:
- миграцию/DDL для `stock_daily_snapshots`
- модуль `src/capture_stock_snapshot.py`
- команду запуска через `src.main --mode stock_snapshot`
- и потом подключить ежедневный запуск по расписанию
