# Тумблер «План производства» в отчёте «Остатки»

**Дата:** 2026-04-17
**Статус:** дизайн утверждён пользователем, готов к планированию реализации

## Цель

Дать пользователю в отчёте «Остатки» быстро переключиться на представление «План производства», которое по остаткам и скорости продаж показывает какие артикулы нужно запустить в производство, с какой срочностью и в каком количестве.

## Контекст

- Отчёт `stock_balances` рендерится функцией `renderStockBalances()` в [web/orders_dashboard.html:9057](web/orders_dashboard.html#L9057). Контейнер — `#stock_balances_wrap`.
- Endpoint `/api/stock-balances` отдаёт `items` с offer_id, name, total, fbo_total, fbs_total, cluster/warehouse detail. Скорости продаж и рекомендаций производства в ответе нет.
- Endpoint `/api/supply-plan` (хендлер `get_supply_plan` в [src/dashboard/routes/supply.py](src/dashboard/routes/supply.py)) отдаёт для каждого артикула `avg_daily_sales`, `depletion_days_left`, `turnover_grade`, `recommended_supply`, полный стек остатков (fbo_available, fbo_supply, fbo_transit, fbo_acceptance, fbs).
- Мэтчинг между ответами — по `offer_id`.

## UX

### Тумблер-кнопка

В хедере отчёта «Остатки» (над существующими sum-карточками) появляется сегментированный тумблер:

```
[ 📦 Остатки ] [ 🏭 План производства ]
```

Активная вкладка подсвечена. Состояние хранится в `#stock_balances_wrap.dataset.view` (значения `stocks` | `production`), начальное — `stocks`.

### Вид «План производства»

**Шапка панели:**
- Поле ввода «Горизонт производства, дней» — `<input type="number">` с default `60`, min `7`, max `365`.
- Кнопка сброса к 60 (опционально).
- 4 сводных карточки:
  1. 🔴 Критично — `N артикулов / M ед.`
  2. 🟠 Срочно — `N / M`
  3. 🟡 В плане — `N / M`
  4. Итого к производству — `M ед.` (сумма по всем приоритетам, БЕЗ 🟢 и ⚪)

**Таблица**, сгруппированная по приоритету. Каждая группа — сворачиваемая секция с иконкой и названием. Группы 🟢/⚪ свёрнуты по умолчанию.

Колонки: `Артикул | Название | v/день | Остаток | ≈дней | К произв. | Статус Ozon`

Стили строк: мягкая фоновая подсветка в тон цвета группы (`#fff0f0` / `#fff5e6` / `#fffae0` / `#f2fbf2` / `#f5f5f5`).

### Приоритизация (по **текущей** дорожке остатка, а не горизонту)

- `days_left = stock / v` если v > 0, иначе ∞
- `v = 0` → ⚪ Без продаж (свёрнуто)
- `days_left ≤ 1` → 🔴 Критично
- `1 < days_left ≤ 14` → 🟠 Срочно
- `14 < days_left ≤ 35` → 🟡 В плане
- `days_left > 35` → 🟢 Запас OK (свёрнуто)

### Определение «Остаток»

В колонке таблицы и формулах используется единое значение:

```
stock = fbo_available + fbo_supply + fbo_transit + fbo_acceptance + fbs
```

Данные берём из supply-plan item (поля `stock_fbo_*`, `stock_fbs`). Это включает всё, что уже в пути к Ozon и в приёмке — то есть «реальный будущий запас» к моменту производства.

### Формула «К производству»

```
production_qty = max(0, ceil(horizon_days × v/день) − stock)
```

Где `horizon_days` — значение поля горизонта (default 60), `v/день` — `avg_daily_sales` из supply-plan.

Для артикулов с `v = 0` (🟢/⚪) поле «К произв.» показывает `—`.

## Данные и поток

1. При выборе отчёта «Остатки» и нажатии «Показать» фронт параллельно запрашивает:
   - `GET /api/stock-balances?...` (как сейчас)
   - `GET /api/supply-plan` (новое, без параметров)
2. Оба ответа кэшируются в замыкании (переменные `stockBalancesData`, `supplyPlanMap`).
3. `supplyPlanMap` — Map `offer_id → supplyItem`.
4. Рендер — по текущему `dataset.view`:
   - `stocks` → `renderStockBalances(stockBalancesData)` (существует, не трогаем)
   - `production` → новая функция `renderProductionPlan(stockBalancesData, supplyPlanMap, horizonDays)`
5. Клик по тумблеру только меняет `dataset.view` и перерисовывает.
6. Изменение поля «Горизонт» — перерисовка только production view (пересчёт «К произв.» и карточек).

### Обработка артикулов, которых нет в supply-plan

Артикулы из stock-balances без соответствия в supply-plan:
- `v/день` = 0
- попадают в группу ⚪ Без продаж
- «К произв.» = `—`

Это нормально: supply-plan возвращает только артикулы с продажами/остатками за окно. Новые или экзотические SKU — в серую группу.

## Бэкенд

**Без изменений.** Всё считается в JS на фронте. `/api/supply-plan` уже готов.

## Что не делаем (YAGNI)

- Не добавляем lead-time производства в формулу (можно добавить поле позже, рядом с горизонтом).
- Не экспортируем план в Excel (сейчас только на экран).
- Не сохраняем пользовательский горизонт между сессиями (значение в полях, не в localStorage).
- Не меняем API и БД.

## Тест-кейсы после реализации

1. Открыть отчёт «Остатки», тумблер должен быть на «📦 Остатки», таблица как прежде.
2. Переключить на «🏭 План производства» → подгрузка `/api/supply-plan` (в Network), рендер с группировкой, цифры совпадают с моим ответом в чате для текущих остатков.
3. Сменить горизонт с 60 на 30 → «К произв.» и «Итого» уменьшаются, группировка не меняется.
4. Сменить на 120 → цифры растут.
5. Переключение обратно на «Остатки» → отдача мгновенная, без повторных запросов.
6. Артикулы из 🟢 / ⚪ скрыты; клик по заголовку группы — разворачивает.
