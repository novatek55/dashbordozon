# Диагностика SKU — график «CR корзина → заказ»

**Дата:** 2026-04-17
**Статус:** дизайн утверждён пользователем, готов к планированию реализации

## Цель

Добавить в блок «Диагностика CR» карточку-график конверсии из корзины в заказ, чтобы увидеть конечный этап воронки (просмотры → корзина → **заказ**). Сейчас виден только CR в корзину, а финальная конверсия «положил, но не купил» (цена / сроки доставки) невидима.

## Контекст

- Блок «Конверсия в корзину по дням» строится функцией, начинающейся на [web/orders_dashboard.html:5551](web/orders_dashboard.html#L5551). Формула: `hits_tocart / session_view_pdp * 100`, рендер — столбцы + Bollinger/pctl через `globalTsStatsLine` и `globalTsBollingerSvg`.
- В каждой daily-точке (`item.daily[i]`) уже есть и `hits_tocart`, и `ordered_units` — бэкенд трогать не нужно (пример использования обоих полей в summary: [web/orders_dashboard.html:5027](web/orders_dashboard.html#L5027)).
- Стилистика и размеры карточки-графика стандартизованы: `width=920`, `height=140`, `sku-impressions-chart`, `sku-impressions-title`, `sku-impressions-summary`, легенда, сворачивание через `sku-collapse-btn`, статистика через `globalTsStatsLine`, аномалии через Bollinger/dots, промо-события через `globalPromoEventsSvg`.

## Что меняем в UX

Под карточкой «Конверсия в корзину по дням» появляется новая карточка такого же визуала и размера:

**Заголовок:** «CR корзина → заказ по дням»
**Подзаголовок:** `Средняя за период: X.XX%` (среднее только по дням, где корзины > 0).
**Легенда:** фиолетовый квадратик + «CR корзина → заказ %» (тот же цвет, что у «CR в корзину» — пользователь подтвердил, что читается как продолжение воронки).
**Статистика:** та же строка `Z=... MAD=... IQR=... BB=... ↑/↓%/д pctl XX% N/N` через `globalTsStatsLine(series)`, та же плашка НОРМА/ВНИМАНИЕ.

Столбцы — такого же цвета, что у соседнего графика «CR в корзину» (фиолетовый `#a78bdb` / label `#5b3a99` — использовать те же переменные `barColor`/`labelColor` из исходной функции).

## Формула и обработка нулей

```
cart = p.hits_tocart
orders = p.ordered_units
cr = cart > 0 ? (orders / cart) * 100 : null
```

Поведение при `cart == 0` (корзина была пустой, CR не определён):

- В `convOrderSeries` храним `null` для таких дней (не 0).
- **Размерность оси X и количество столбцов полностью совпадают с соседними графиками** — дни на месте, просто столбец в этот день не рисуется (высота 0, label пустой).
- Такие дни **исключены** из расчёта среднего за период и из статистики (`globalTsStatsLine`, Bollinger, MAD, IQR, pctl). Ноль не занижает аномалии.
- Для функций стат-обработки, которые ожидают массив чисел, передаём отфильтрованную серию (`series.filter(v => v != null)`) на уровне статистики. Для отрисовки столбцов и позиционирования — полная серия с `null`.
- Значения > 100% возможны (заказ оформлен из корзины прошлого дня). Верхнюю границу Y оставляем по `Math.max(0.5, ...convOrderSeries.filter(v => v != null))` — как у соседнего графика, но игнорируем `null`.

## Размещение в DOM

Порядок карточек в блоке диагностики (сверху вниз):

1. Клики / показы / impressions (существующие)
2. **Конверсия в корзину по дням** (существующий)
3. **CR корзина → заказ по дням** ← новое, сразу после (2)
4. Средняя позиция по дням (существующий)
5. Остальные карточки

Функция вызывается в той же точке, где сейчас генерируется карточка CR в корзину, сразу следующим `push`/конкатенацией.

## Реализация (frontend-only)

- Новая функция `buildCartToOrderChart(item, chartId)` в [web/orders_dashboard.html](web/orders_dashboard.html), рядом с функцией CR в корзину (~строка 5540).
- Сигнатура и структура — копия функции «Конверсия в корзину» с тремя отличиями:
  1. Формула серии: `orders / cart * 100` с null-для-нулевых-корзин.
  2. Заголовок, summary, id, подпись легенды.
  3. Для `globalTsStatsLine`, `globalTsBollingerSvg`, `globalTsBollingerDots`, `avgConv` — использовать отфильтрованную серию; для отрисовки столбцов и xTicks — исходную (с null).
- Вызов новой функции добавить сразу после вызова `buildCartConversionChart(...)` (или как она реально называется) в рендер-цепочке карточек диагностики.

**Пустые данные**: если во всём периоде `hits_tocart == 0` → показать `<div class="analytics-chart-empty">Нет корзин за период</div>`, аналогично пустому кейсу соседних графиков.

## Бэкенд

Без изменений. Все нужные поля уже в `/api/sku-diagnostics` daily-точках.

## Что НЕ делаем (YAGNI)

- Не добавляем вторую метрику на существующий график (отдельный блок — вариант №3 из брейнсторма).
- Не вводим «окно сопоставления корзина↔заказ по дате». Считаем в дне как есть — при стабильном потоке усреднение за период всё выправит.
- Не добавляем переключатель «от корзины / от начала воронки». При необходимости сквозной CR показов→заказ позже можно добавить отдельным блоком.
- Не меняем существующий блок «Воронка» (summary) и общий stacked-chart — там CR корзина→заказ уже считается как summary-число.

## Тест-кейсы после реализации

1. Открыть «Диагностика SKU» для любого SKU с продажами → под «Конверсия в корзину по дням» появляется «CR корзина → заказ по дням» такого же размера.
2. Кол-во столбцов/xTicks/ширина идентичны соседнему графику (визуальная проверка — края выровнены).
3. Для SKU, где в середине периода был день с 0 корзин, проверить: в новом графике этот день — пустой столбец (не 0%), а в среднем за период и в pctl этот день не учитывается.
4. Plашка НОРМА/ВНИМАНИЕ и строка Z/MAD/IQR/BB/pctl заполнены и соответствуют серии без нулей.
5. Промо-события (вертикальные маркеры) отображаются так же, как у соседних графиков.
6. SKU без корзин за весь период → «Нет корзин за период».
7. SKU с заказом > корзин в один день (CR > 100%) — столбец и label рисуются корректно, верхняя граница Y расширяется.
