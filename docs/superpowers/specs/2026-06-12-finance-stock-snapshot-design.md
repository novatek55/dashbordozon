# Дизайн: snapshot-таблица для финансовых данных отчета "Остатки"

## Цель

Нужно сохранить в отдельной таблице данные из отчета "Остатки", чтобы другой проект мог забирать:

- остатки по артикулу;
- сумму потенциального начисления по артикулу;
- итоговую агрегированную строку по всему срезу.

Источник должен обновляться вместе с текущим обновлением остатков и давать стабильный снимок на момент синхронизации.

## Решение

Добавляется отдельная snapshot-таблица `finance_stock_snapshots`, которая хранит:

- строки уровня артикула (`row_type = 'item'`);
- одну итоговую строку по всему срезу (`row_type = 'total'`).

Таблица не заменяет существующие `analytics_stocks` и `fbs_warehouse_stocks`, а строится поверх них как производный слой, аналогично существующему паттерну `stock_daily_snapshots`.

## Почему отдельная таблица

Рассматривались три варианта:

1. Отдельная snapshot-таблица.
2. SQL view/materialized view.
3. Расширение существующих stock-таблиц.

Выбран вариант 1, потому что он:

- дает стабильный срез "на момент обновления";
- удобен для чтения из другого проекта без дополнительной агрегации;
- позволяет хранить историю по датам;
- не смешивает сырой источник остатков и производный финансовый расчет.

## Структура данных

Новая таблица `finance_stock_snapshots` должна содержать поля:

- `id`
- `snapshot_date`
- `snapshot_at`
- `row_type` (`item` или `total`)
- `offer_id`
- `sku`
- `name`
- `stock_total`
- `stock_fbo`
- `stock_fbs`
- `potential_revenue`
- `potential_accrual`
- `created_at`

Ожидаемое поведение:

- для `row_type = 'item'` заполняются данные по конкретному артикулу;
- для `row_type = 'total'` хранятся суммарные значения по всему срезу;
- `offer_id`, `sku`, `name` для итоговой строки могут быть `NULL` или иметь служебное имя, но итоговая строка должна однозначно определяться по `row_type = 'total'`.

Рекомендуемое ограничение уникальности:

- одна запись на комбинацию `snapshot_date + row_type + offer_id`;
- для итоговой строки допустим `offer_id = NULL`, поэтому итоговую запись лучше вставлять отдельно с проверкой по `row_type = 'total'`, либо использовать дополнительный служебный ключ.

На этапе реализации предпочтительно сделать более надежный ключ:

- `snapshot_date + row_type + entity_key`,

где `entity_key`:

- для товарной строки равен нормализованному `offer_id`;
- для итоговой строки равен `'__total__'`.

Это избавит от проблем с `NULL` в unique-индексах.

## Источник значений

Поля остатков берутся из тех же источников, что и текущий отчет "Остатки":

- `analytics_stocks` как источник FBO-остатков;
- `fbs_warehouse_stocks` как источник FBS-остатков.

Агрегация выполняется по артикулу.

Расчетные поля:

- `stock_fbo` — агрегированный остаток FBO по артикулу;
- `stock_fbs` — агрегированный остаток FBS по артикулу;
- `stock_total = stock_fbo + stock_fbs`;
- `potential_accrual` — значение, совпадающее с колонкой "Потенц. начислено" в текущем отчете "Остатки";
- `potential_revenue` — значение, совпадающее с колонкой "Потенц. выручка" в текущем отчете "Остатки".

Ключевое требование: значения `potential_accrual` и `potential_revenue` должны рассчитываться из той же общей логики, что используется в UI отчета "Остатки", без дублирования и без альтернативной формулы в новом месте.

## Точка обновления

После успешного обновления stock-данных должен запускаться отдельный шаг snapshot-сохранения:

1. собрать актуальный срез по артикулам;
2. рассчитать финансовые поля по текущей логике отчета;
3. удалить/перезаписать записи за тот же `snapshot_date`, если sync запускается повторно в тот же день;
4. вставить строки `item`;
5. вставить строку `total`.

Базовая точка интеграции:

- после `sync_analytics_stocks`;
- с учетом того, что итоговые данные должны использовать также актуальные `fbs_warehouse_stocks`.

На этапе реализации допустимы два варианта:

1. вызывать snapshot-запись в конце `sync_analytics_stocks`, если FBS-остатки уже актуальны;
2. вынести snapshot-запись в общий orchestration-шаг после последовательного обновления `analytics_stocks` и `fbs_warehouse_stocks`.

Предпочтителен вариант 2, так как он лучше гарантирует согласованность FBO и FBS в одном срезе.

## Переиспользование логики

Чтобы не допустить расхождения цифр между интерфейсом и новой таблицей:

- общую сборку данных отчета "Остатки" нужно вынести в переиспользуемую функцию, сервис или SQL-builder;
- UI маршрута "Остатки" и snapshot-запись должны использовать один и тот же источник расчета.

Нельзя поддерживать две независимые реализации формулы `potential_accrual` и `potential_revenue`.

## API для другого проекта

Для чтения данных добавляются отдельные API-роуты:

- `/api/finance-stock-snapshots/latest`
- `/api/finance-stock-snapshots?date=YYYY-MM-DD`

Ожидаемое поведение:

- `latest` возвращает последний доступный snapshot;
- `date` возвращает snapshot за указанную дату;
- в ответе должны присутствовать и `item`-строки, и `total`-строка, либо `total` может возвращаться отдельным полем верхнего уровня, если это упростит потребление.

Предпочтение на этапе реализации:

- вернуть JSON вида `{ snapshot_date, snapshot_at, total, items }`,

так как это удобнее для внешнего проекта, чем плоский список со смешанными типами строк.

## Изменения в коде

Минимальный набор изменений:

1. Добавить новую модель SQLAlchemy в `src/models.py`.
2. Добавить таблицу в `USED_TABLES` в `src/database.py`.
3. Добавить метод snapshot-записи в `src/sync_manager.py`.
4. Вынести общий расчет данных отчета "Остатки" в переиспользуемый слой.
5. Добавить API-роут(ы) чтения snapshot-данных.
6. Добавить тесты на структуру и содержимое snapshot-данных.

## Тестирование

Нужно покрыть как минимум:

- формирование строк `item` из FBO/FBS-источников;
- корректный расчет `stock_total`;
- наличие и корректность строки `total`;
- идемпотентную перезапись snapshot-а за ту же дату;
- совпадение `potential_accrual` и `potential_revenue` с общей логикой отчета "Остатки".

Если текущая логика отчета плохо тестируется напрямую, сначала нужно выделить чистую функцию/сервис агрегации, а затем писать тесты на нее.

## Риски

- расхождение между UI отчета и snapshot-таблицей, если формула будет продублирована;
- несогласованный срез FBO/FBS, если snapshot писать до завершения обоих обновлений;
- нестабильный unique-ключ, если использовать `NULL`-значения напрямую.

## Границы задачи

В рамках этой задачи не требуется:

- менять визуальное представление отчета "Остатки";
- переписывать существующие таблицы источников;
- менять формулу расчета потенциальной выручки или потенциального начисления.

Нужно только выделить, сохранить и отдать уже существующие данные в удобном snapshot-формате для другого проекта.
