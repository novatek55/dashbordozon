# WB Finance Report - Этап 1: метод получения данных

Дата фиксации: 2026-05-07

## Цель этапа

Определить стабильный способ получения данных WB для финансового отчета и зафиксировать резервный сценарий на случай изменений API.

## Решение по источникам данных

Основной источник (primary):

- API-группа `financial-reports-and-accounting` в `finance-api.wildberries.ru`
- Метод детализации за период:
  - `POST /api/finance/v1/sales-reports/detailed`
  - Назначение: построчная детализация продаж/возвратов и финансовых сумм по периоду.
  - Пагинация: через `rrdId` (запрос стартует с `rrdId=0`, далее берем `rrdId` последней строки, повторяем до `204 No data`).

Дополнительный источник (для сверки и метаданных отчетов):

- `POST /api/finance/v1/sales-reports/list`
  - Используем для списка отчетов (daily/weekly), проверки полноты периода, выбора `reportId` для точечной отладки.

Резервный источник (fallback, если недоступен primary):

- `GET /api/v5/supplier/reportDetailByPeriod` (statistics-api, legacy)
  - Использовать только как временный fallback.
  - В ETL логировать факт переключения на fallback.

## Почему так

- В новой документации WB финансовые отчеты централизованы в `finance-api`.
- Для финотчета за период нужен детальный поток строк; его дает `sales-reports/detailed`.
- `rrdId`-инкремент позволяет догружать большие периоды без потери строк.

## Контракт загрузки (v1)

- Авторизация: `HeaderApiKey` (токен категории Finance).
- Период: передаем RFC3339, используем МСК (`UTC+3`) для границ суток.
- Периодичность:
  - `period=daily` для управленческого отчета по дням.
  - `period=weekly` только для сверки с weekly-срезами WB.
- Размер порции: `limit=100000` (или ниже при фактических ограничениях/таймаутах).
- Окончание чтения: ответ `204`.
- Идемпотентность: уникальный ключ в raw-слое = (`dateFrom`,`dateTo`,`rrdId`,`nmId`,`srid`) либо `hash(row_json)` при отсутствии устойчивого составного ключа.

## Минимальный PoC для следующего шага

1. Запрос `sales-reports/detailed` за 1 день с `rrdId=0`.
2. Цикл догрузки до `204`.
3. Сохранение сырых строк в staging-таблицу WB (без трансформаций).
4. Логирование:
   - период,
   - число строк,
   - последний `rrdId`,
   - источник (`finance-v1` или `legacy-v5`),
   - длительность и ошибки.

## Критерии готовности этапа 1

- Зафиксирован primary endpoint и fallback endpoint.
- Описан протокол пагинации/инкремента (`rrdId` + `204`).
- Описаны правила периода и таймзоны (МСК).
- Есть PoC-сценарий для перехода к этапу 2 (загрузка в БД).

## Риски и контроль

- Риск: региональные/аккаунтные ограничения некоторых методов.
  - Контроль: тестовый прогон `sales-reports/list` и `sales-reports/detailed` конкретным токеном.
- Риск: изменение контракта полей.
  - Контроль: хранить raw JSON полностью и версионировать нормализацию.
- Риск: лимит 1 запрос/мин на часть методов.
  - Контроль: встроенный rate limiter + backoff при `429`.
