## 2) Рабочие статьи для нашего WB-отчета (на русском)

1. `Валовая выручка`  
Источник: `Вайлдберриз реализовал Товар (Пр)` (`P`)

2. `Комиссия маркетплейса`  
Источник: `Вознаграждение Вайлдберриз (ВВ), без НДС` (`AF`)  
Примечание: обычно отрицательная сумма (расход).

3. `Логистика прямая`  
Источник: `Услуги по доставке товара покупателю` (`AK`)

4. `Логистика обратная`  
Источник: строки/поля возвратной логистики (в текущем API-плоском срезе `returnAmount`, если есть).

5. `Эквайринг / платёжные услуги`  
Источник: `Компенсация платёжных услуг/Комиссия за интеграцию платёжных сервисов` (`AC`)

6. `Штрафы`  
Источник: `Общая сумма штрафов` (`AO`)

7. `Прочие удержания`  
Источник (суммарно): `Удержания` (`BI`) + `Хранение` (`BH`) + `Операции на приемке` (`BJ`) + иные сервисные удержания.

8. `К выплате`  
Источник: `К перечислению Продавцу за реализованный Товар` (`AH`)

## 3) Маппинг на текущие ключи витрины `wb_finance_daily`

- `gross_revenue` -> `Валовая выручка`
- `marketplace_commission` -> `Комиссия маркетплейса` (из поля API `vw`)
- `logistics_direct` -> `Логистика прямая` (из поля API `deliveryService`)
- `logistics_reverse` -> `Логистика обратная`
- `acquiring` -> `Эквайринг / платёжные услуги`
- `penalties` -> `Штрафы`
- `other_deductions` -> `Прочие удержания` (из `deduction + paidStorage + paidAcceptance`)
- `to_pay` -> `К выплате`

## 4) Важное замечание по знакам

В WB-файлах часть расходов приходит со знаком минус (например, комиссия), часть как положительные значения в отдельных колонках расходов.  
В нашем отчете храним “как пришло из API/витрины” и отображаем отдельными статьями, а финальный P&L считаем формулой в следующем слое отчета.
